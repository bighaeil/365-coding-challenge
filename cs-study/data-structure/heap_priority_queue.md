# 힙(Heap)과 우선순위 큐(Priority Queue)

## 질문

힙(Heap)이란 무엇이고, 우선순위 큐(Priority Queue)와 어떤 관계인가요?
내부 구조, 주요 연산의 시간 복잡도, 그리고 실전에서 어디에 쓰이는지 설명하세요.

---

## 핵심 답변

### 1. 기본 개념

**힙(Heap)** 은 **완전 이진 트리(Complete Binary Tree)** 를 기반으로 한 자료구조로,
부모와 자식 사이에 **특정한 대소 관계(힙 속성)** 를 유지합니다.

- **최대 힙(Max Heap)**: 부모 ≥ 자식 → 루트가 항상 **최댓값**
- **최소 힙(Min Heap)**: 부모 ≤ 자식 → 루트가 항상 **최솟값**

> ⚠️ 주의: 힙은 **완전히 정렬된 구조가 아닙니다.**
> "루트가 최대/최소"라는 것만 보장되며, 형제 간에는 순서가 없습니다.

**우선순위 큐(Priority Queue)** 는 "우선순위가 높은 것부터 꺼내는 큐"라는 **추상 자료형(ADT)** 이고,
**힙은 그것을 가장 효율적으로 구현하는 자료구조**입니다.

> 즉, **우선순위 큐 = 개념(What), 힙 = 구현(How)**

---

### 2. 내부 구조 — 배열로 구현하는 완전 이진 트리

힙은 포인터 기반 트리가 아니라 **1차원 배열**로 구현합니다.
완전 이진 트리이기 때문에 배열 인덱스만으로 부모/자식 관계가 계산됩니다.

인덱스를 1부터 시작할 때:
- 부모: `i / 2`
- 왼쪽 자식: `i * 2`
- 오른쪽 자식: `i * 2 + 1`

인덱스를 0부터 시작할 때:
- 부모: `(i - 1) / 2`
- 왼쪽 자식: `2 * i + 1`
- 오른쪽 자식: `2 * i + 2`

예시 (Max Heap):

```
          [50]
         /    \
      [30]    [40]
      /  \    /
   [10] [20] [35]

배열: [50, 30, 40, 10, 20, 35]
```

---

### 3. 핵심 연산

#### 🔹 삽입 (Insert) — O(log n)

1. 배열 맨 끝에 새 값을 추가 (트리의 마지막 자리)
2. **부모와 비교**하며 힙 속성을 만족할 때까지 위로 올림 → **Up-Heap / Sift-Up**

#### 🔹 삭제 (Extract-Max / Extract-Min) — O(log n)

1. 루트를 꺼낸다(= 결과값)
2. 배열 **마지막 원소를 루트로 이동**
3. **자식과 비교**하며 힙 속성을 만족할 때까지 아래로 내림 → **Down-Heap / Sift-Down**

#### 🔹 최댓값/최솟값 조회 (Peek) — O(1)

루트 = `heap[0]` 을 바로 반환.

#### 🔹 힙 생성 (Build Heap) — O(n)

배열을 한 번에 힙으로 만드는 연산. 하나씩 삽입하면 O(n log n)이지만,
**마지막 비-리프 노드부터 거꾸로 Sift-Down** 하면 **O(n)** 에 만들 수 있습니다.

---

### 4. 시간 복잡도 요약

| 연산            | 힙 (Heap) | 정렬 배열 | 정렬 안 된 배열 |
| --------------- | --------- | --------- | --------------- |
| Peek (최댓값)   | **O(1)**  | O(1)      | O(n)            |
| Insert          | **O(log n)** | O(n)   | O(1)            |
| Extract-Max     | **O(log n)** | O(1)   | O(n)            |
| 임의 검색       | O(n)      | O(log n)  | O(n)            |
| Build (초기 생성)| **O(n)** | O(n log n)| O(n)            |

**힙의 강점**: 삽입과 "최댓값 꺼내기"의 **균형** 이 가장 좋습니다.
"가장 큰(또는 작은) 것만 빨리 꺼내면 되는" 시나리오에 최적.

---

### 5. 실전 활용

#### ① 우선순위 큐 구현
운영체제의 **프로세스 스케줄링**, 네트워크 패킷 처리, 프린터 대기열(ex15) 등.

#### ② 힙 정렬 (Heap Sort)
배열을 힙으로 만든 뒤 계속 루트를 뽑으면 정렬됨.
- 시간 복잡도: **O(n log n)** (항상 보장)
- 공간 복잡도: **O(1)** (in-place)
- 단점: 퀵 정렬보다 캐시 지역성이 떨어져 실측은 더 느린 편

#### ③ Top-K 문제 (면접 단골 ⭐)
"배열에서 가장 큰 K개를 구하시오"
- **크기 K의 최소 힙**을 유지하며 순회
- 힙 크기보다 큰 값을 만나면 root(최솟값) 제거 후 삽입
- 시간 복잡도: **O(n log k)** (전체 정렬 O(n log n)보다 빠름)

#### ④ 다익스트라(Dijkstra) 최단 경로
"현재 가장 거리가 짧은 노드"를 매번 꺼내야 함 → 최소 힙으로 O((V+E) log V).

#### ⑤ 중앙값 스트리밍 (Median in a Stream)
**최대 힙 + 최소 힙**을 같이 사용해 실시간 중앙값을 O(log n)에 유지.

#### ⑥ 이벤트/작업 스케줄러
"다음 실행 시간이 가장 가까운 작업"을 최소 힙으로 관리.

## 예상 면접 질문

**Q1. 힙과 이진 탐색 트리(BST)의 차이는 무엇인가요?**
> BST는 **모든 노드 간 전순서**(왼쪽 < 부모 < 오른쪽)를 보장해 임의 검색이 O(log n)입니다.
> 힙은 **부모-자식 간 대소 관계만** 보장하므로 임의 검색은 O(n)이지만, 최대/최소 조회는 O(1)이고 삽입·삭제가 항상 O(log n)으로 안정적입니다.
> "정렬된 탐색"이 필요하면 BST, "최대/최소만 빠르게"가 필요하면 힙이 적합합니다.

**Q2. 힙은 왜 배열로 구현하나요?**
> 완전 이진 트리이기 때문에 인덱스 계산만으로 부모/자식 관계를 알 수 있어 **포인터가 불필요**하고,
> 배열은 **메모리 지역성이 좋아 캐시 효율**도 뛰어납니다.

**Q3. Build Heap이 왜 O(n)인가요? O(n log n)이 아닌가요?**
> 하나씩 Sift-Up으로 삽입하면 O(n log n)이지만,
> Sift-Down 기반으로 마지막 비-리프부터 거꾸로 쌓으면 대부분의 노드(리프 쪽)가 적은 횟수만 이동하므로
> 수학적으로 합산하면 **O(n)** 이 됩니다.

**Q4. 가장 큰 K개를 구할 때 정렬(O(n log n)) 대신 힙(O(n log k))을 쓰는 이유는?**
> K가 n보다 훨씬 작을 때 **log k < log n** 이므로 더 빠릅니다.
> 또한 **스트리밍 데이터** 처럼 전체를 한 번에 볼 수 없는 경우에도 적용 가능합니다.

**Q5. 힙 정렬은 O(n log n)인데 왜 실무에선 퀵 정렬을 더 많이 쓰나요?**
> 힙 정렬은 노드 이동이 배열 전반을 뛰어다녀 **캐시 미스가 많습니다.**
> 퀵 정렬은 분할 후 지역적으로 동작해 평균적으로 상수 계수가 더 작아 실측이 빠릅니다.
> 다만 최악이 O(n²)이라는 단점이 있어, 상황에 따라 둘을 조합한 **Introsort** 가 쓰입니다(C++ `std::sort`).

- **"힙은 정렬된 구조다"** 는 흔한 오개념입니다. → "부모-자식 관계만 보장"이라고 명확히 구분하세요.
