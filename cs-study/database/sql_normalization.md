# SQL 정규화 (Normalization)

## 질문

데이터베이스 정규화란 무엇이고, 1NF·2NF·3NF가 각각 무엇을 해결하는지 설명하세요.
정규화의 장단점과 실무에서 비정규화(Denormalization)를 선택하는 기준도 설명하세요.

---

## 핵심 답변

### 1. 정규화란?

**정규화(Normalization)** 는 데이터베이스 테이블을 설계할 때 **중복을 제거하고 이상 현상(Anomaly)을 방지**하기 위해 테이블을 쪼개는 과정입니다.

#### 이상 현상(Anomaly) — 정규화가 필요한 이유

| 이상 현상 | 설명 | 예시 |
|----------|------|------|
| **삽입 이상** | 불필요한 데이터 없이 삽입 불가 | 수강생 없는 강좌를 등록할 수 없음 |
| **갱신 이상** | 일부만 수정 시 불일치 발생 | 강사 이름을 한 행만 수정해 데이터 불일치 |
| **삭제 이상** | 삭제 시 의도치 않은 데이터 손실 | 마지막 수강생 삭제 시 강좌 정보도 사라짐 |

---

### 2. 제1정규형 (1NF) — 원자값

**규칙: 모든 컬럼의 값이 원자값(Atomic Value)이어야 한다.**  
→ 하나의 셀에 여러 값이 들어가면 안 됩니다.

#### 위반 예시

| 학생ID | 이름 | 수강과목 |
|--------|------|---------|
| 1 | 철수 | 수학, 영어, 과학 |  ← 한 셀에 여러 값 ❌

#### 1NF 적용 후

| 학생ID | 이름 | 수강과목 |
|--------|------|---------|
| 1 | 철수 | 수학 |
| 1 | 철수 | 영어 |
| 1 | 철수 | 과학 |

> 반복 그룹 제거 + 각 행이 유일하게 식별 가능(기본키 존재)

---

### 3. 제2정규형 (2NF) — 부분 함수 종속 제거

**규칙: 1NF를 만족하고, 기본키의 일부에만 종속된 컬럼이 없어야 한다.**  
→ 복합 기본키일 때, 키의 일부에만 의존하는 컬럼을 분리합니다.

#### 위반 예시 (기본키: 학생ID + 과목코드)

| 학생ID | 과목코드 | 학점 | 과목명 | 담당교수 |
|--------|---------|------|--------|---------|
| 1 | CS101 | A | 자료구조 | 김교수 |
| 2 | CS101 | B | 자료구조 | 김교수 |  ← 과목명, 담당교수는 과목코드에만 종속 ❌

```
학생ID + 과목코드 ──► 학점       (완전 함수 종속 ✅)
과목코드          ──► 과목명     (부분 함수 종속 ❌)
과목코드          ──► 담당교수   (부분 함수 종속 ❌)
```

#### 2NF 적용 후 — 테이블 분리

**수강 테이블** (기본키: 학생ID + 과목코드)

| 학생ID | 과목코드 | 학점 |
|--------|---------|------|
| 1 | CS101 | A |
| 2 | CS101 | B |

**과목 테이블** (기본키: 과목코드)

| 과목코드 | 과목명 | 담당교수 |
|---------|--------|---------|
| CS101 | 자료구조 | 김교수 |

---

### 4. 제3정규형 (3NF) — 이행 함수 종속 제거

**규칙: 2NF를 만족하고, 기본키가 아닌 컬럼이 다른 비키 컬럼에 종속되지 않아야 한다.**  
→ A → B → C 처럼 거쳐가는 종속을 제거합니다.

#### 위반 예시 (기본키: 학생ID)

| 학생ID | 이름 | 학과코드 | 학과명 | 학과위치 |
|--------|------|---------|--------|---------|
| 1 | 철수 | CS | 컴퓨터공학과 | 공학관 |

```
학생ID ──► 학과코드            (직접 종속 ✅)
학과코드 ──► 학과명, 학과위치  (이행 종속 ❌)
학생ID ──► 학과명              (이행 함수 종속)
```

학과명, 학과위치는 기본키(학생ID)가 아닌 학과코드에 종속됨.  
→ 학과코드가 바뀌면 학과명도 따로 수정해야 하는 갱신 이상 발생.

#### 3NF 적용 후 — 테이블 분리

**학생 테이블**

| 학생ID | 이름 | 학과코드 |
|--------|------|---------|
| 1 | 철수 | CS |

**학과 테이블**

| 학과코드 | 학과명 | 학과위치 |
|---------|--------|---------|
| CS | 컴퓨터공학과 | 공학관 |

---

### 5. 정규형 단계 요약

```
원본 테이블
    │
    ▼ 원자값 보장 (반복 그룹 제거)
  1NF
    │
    ▼ 부분 함수 종속 제거 (복합키 → 분리)
  2NF
    │
    ▼ 이행 함수 종속 제거 (비키 → 비키 종속 분리)
  3NF   ← 실무에서 보편적 목표
    │
    ▼ 결정자가 후보키인지 확인
  BCNF  (3NF의 강화 버전, 실무에서 필요시 적용)
```

---

### 6. 정규화의 장단점

| | 장점 | 단점 |
|-|------|------|
| **정규화** | 중복 제거, 데이터 일관성, 저장 공간 절약 | JOIN 증가 → 읽기 성능 저하 |
| **비정규화** | JOIN 감소 → 읽기 성능 향상 | 중복 발생, 갱신 이상 위험 |

---

### 7. 비정규화(Denormalization)를 선택하는 기준

**읽기가 압도적으로 많고(Read-heavy), 쓰기는 드물며, 성능이 중요한 경우.**

```sql
-- 정규화된 경우: 매 조회마다 JOIN 필요
SELECT o.id, u.name, u.email, p.title, p.price
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id;

-- 비정규화: orders에 user_name, product_title을 중복 저장
-- → JOIN 없이 단일 테이블 조회 (빠르지만 데이터 중복)
SELECT id, user_name, product_title, price FROM orders;
```

실무 비정규화 예시:
- 통계/집계 컬럼 미리 저장 (댓글 수, 좋아요 수)
- 자주 함께 조회되는 컬럼을 같은 테이블에 복사
- 데이터 웨어하우스(OLAP)에서는 비정규화된 Star Schema 사용

## 예상 면접 질문

**Q1. 정규화를 왜 하나요?**
> 데이터 중복으로 인한 **삽입·갱신·삭제 이상(Anomaly)** 을 방지하기 위해서입니다. 한 곳만 수정해도 전체 데이터가 일관성을 유지하도록 설계합니다.

**Q2. 2NF와 3NF의 차이를 한 문장으로 설명하세요.**
> 2NF는 **기본키의 일부에 종속**된 컬럼을 제거하고(복합키 상황), 3NF는 **기본키가 아닌 컬럼에 종속**된 컬럼을 제거합니다(이행 종속).

**Q3. 정규화를 끝까지 하면 무조건 좋은 건가요?**
> 아닙니다. 정규화가 심화될수록 테이블이 분리되어 **JOIN이 많아지고 읽기 성능이 저하**됩니다. 트래픽이 많은 서비스에서는 3NF까지만 적용하거나, 조회 성능을 위해 의도적으로 **비정규화(Denormalization)** 를 선택합니다.

**Q4. BCNF는 3NF와 무엇이 다른가요?**
> 3NF는 "기본키 → 비키 컬럼" 종속만 허용하지만, BCNF는 더 엄격하게 **모든 결정자(Determinant)가 후보키**여야 합니다. 교수가 한 과목만 담당할 수 있는 상황처럼 3NF를 만족하지만 BCNF를 위반하는 경우가 있습니다.

## 면접 팁

- **"1NF → 2NF → 3NF"의 각 단계가 무엇을 해결하는지** 를 한 문장씩 말할 수 있어야 합니다.
- 실무 경험을 연결할 때: "정규화 설계 후 쿼리 성능 이슈가 생겨 특정 집계 컬럼을 비정규화했다"처럼 **트레이드오프 인식**을 보여주면 좋습니다.
- **"이행 종속(Transitive Dependency)"** 용어를 정확히 사용하면 전문성을 어필할 수 있습니다.
