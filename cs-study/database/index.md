# 인덱스(Index)

## 질문

데이터베이스에서 인덱스(Index)란 무엇이고, 어떻게 동작하나요?
내부 자료구조, 장단점, 그리고 어떤 상황에서 사용하면 좋은지 설명하세요.

---

## 핵심 답변

### 1. 기본 개념

인덱스는 테이블의 특정 컬럼에 대해 **빠른 검색을 가능하게 하는 별도의 자료구조**입니다.
책의 색인(목차)처럼, 원하는 데이터의 위치를 빠르게 찾아줍니다.

- 인덱스가 **없을 때** → Full Table Scan: O(n)
- 인덱스가 **있을 때** → Index Scan: O(log n)

### 2. 내부 자료구조 — B-Tree

대부분의 RDBMS(MySQL, PostgreSQL 등)는 **B-Tree (Balanced Tree)** 인덱스를 사용합니다.

```
           [30]
          /    \
       [10,20] [40,50]
       /  |  \   |  \
     [5] [15] [25] [35] [45,55]
```

- 루트 → 브랜치 → 리프 순으로 탐색
- 항상 균형이 맞아 **최악에도 O(log n)** 보장
- 범위 검색(`BETWEEN`, `>`, `<`)에 강함

> **Hash Index**: 단순 등치(`=`) 검색만 O(1)이지만 범위 검색 불가

### 3. 장단점

| 구분 | 내용 |
|------|------|
| **장점** | SELECT 속도 향상 (특히 WHERE, JOIN, ORDER BY) |
| **단점** | INSERT / UPDATE / DELETE 시 인덱스도 갱신 → 쓰기 성능 저하 |
| **단점** | 추가 저장 공간 필요 |
| **단점** | 카디널리티가 낮은 컬럼(예: boolean)엔 효과 없음 |

### 4. 인덱스를 사용하면 좋은 경우 vs 나쁜 경우

**좋은 경우**
- WHERE 절에 자주 등장하는 컬럼
- JOIN ON 조건 컬럼
- 카디널리티(고유값 수)가 높은 컬럼 (예: email, user_id)
- ORDER BY / GROUP BY 에 쓰이는 컬럼

**나쁜 경우**
- 카디널리티가 낮은 컬럼 (예: 성별, 상태값 Y/N)
- 쓰기(INSERT/UPDATE/DELETE)가 매우 빈번한 테이블
- 데이터 수가 매우 적은 테이블

### 5. 복합 인덱스 (Composite Index)

두 개 이상의 컬럼을 묶은 인덱스입니다.

```sql
CREATE INDEX idx_name ON users (last_name, first_name);
```

**선두 컬럼 규칙**: `last_name`은 단독으로도 인덱스 활용 가능하지만,
`first_name`만으로는 활용 불가 → **왼쪽부터 순서가 중요**

## 예상 면접 질문

**Q1. 인덱스를 사용하면 항상 조회가 빨라지나요?**
> 아니요. 카디널리티가 낮은 컬럼(예: 성별 M/F)은 인덱스를 타더라도 전체의 절반을 스캔해야 하므로, 옵티마이저가 Full Scan을 선택하는 경우도 있습니다. 또한 쓰기가 빈번한 테이블에서는 오히려 성능이 저하될 수 있습니다.

**Q2. 클러스터드 인덱스와 논클러스터드 인덱스의 차이는?**
> **클러스터드 인덱스**는 실제 데이터 행이 인덱스 순서로 물리적으로 정렬되어 저장됩니다. 테이블당 1개만 존재하며, MySQL InnoDB의 Primary Key가 이에 해당합니다.
> **논클러스터드 인덱스**는 별도의 인덱스 구조에 데이터의 포인터(rowid)를 저장하며, 테이블당 여러 개 생성 가능합니다.

**Q3. 인덱스가 있는데도 Full Scan이 발생하는 이유는?**
> 옵티마이저가 비용을 계산했을 때 ①카디널리티가 낮거나, ②조회 데이터 비율이 높거나(20~30% 이상), ③인덱스 컬럼에 함수/연산을 적용했을 때(`WHERE YEAR(created_at) = 2024`) 인덱스를 무시하고 Full Scan을 선택할 수 있습니다.

## 면접 팁

- **"인덱스 자료구조가 뭐예요?"** → B-Tree, 균형 트리, O(log n) 언급
- **"인덱스 단점은?"** → 쓰기 성능 저하, 추가 공간, 카디널리티 낮으면 효과 없음
- **"복합 인덱스에서 주의할 점은?"** → 선두 컬럼 규칙 (Leftmost Prefix Rule)
- **"실무에서 인덱스 확인 방법?"** → `EXPLAIN SELECT ...` 로 실행 계획 확인
