# 트랜잭션(Transaction)이란 무엇이고, ACID 속성에 대해 설명해주세요.

## 질문

데이터베이스의 트랜잭션이 무엇인지 설명하고,
ACID 속성 4가지를 각각 예시와 함께 설명해주세요.

---

## 핵심 답변

### 1. 트랜잭션이란?

데이터베이스에서 **하나의 논리적 작업 단위**를 구성하는 연산의 집합입니다.
모든 연산이 성공하면 **커밋(Commit)**, 하나라도 실패하면 **롤백(Rollback)** 합니다.

```
예시: A가 B에게 10만원 송금

1. A 계좌에서 10만원 차감
2. B 계좌에 10만원 추가

→ 둘 다 성공 → 커밋
→ 1번 성공 후 2번 실패 → 롤백 (A 계좌 원상복구)
```

---

### 2. ACID 속성

#### A — 원자성 (Atomicity)
트랜잭션의 연산은 **모두 성공하거나 모두 실패**해야 합니다. 중간 상태가 없습니다.

```
송금 중 서버가 다운되면?
→ 롤백으로 A 계좌 차감이 취소됨
→ "반만 실행된" 상태가 존재하지 않음
```

#### C — 일관성 (Consistency)
트랜잭션 실행 전후로 데이터베이스는 **항상 일관된 상태**를 유지해야 합니다.

```
A + B의 잔액 합계는 트랜잭션 전후 동일해야 함
제약 조건(NOT NULL, FK 등)을 항상 만족해야 함
```

#### I — 격리성 (Isolation)
동시에 실행되는 트랜잭션은 **서로 영향을 주지 않아야** 합니다.

```
A→B 송금 트랜잭션이 진행 중일 때
C가 A의 잔액을 조회하면?
→ 격리 수준에 따라 차감 전 금액 또는 차감 후 금액을 볼 수 있음
```

**격리 수준 (낮음 → 높음):**
| 수준 | 설명 |
|------|------|
| READ UNCOMMITTED | 커밋 안 된 데이터도 읽음 (Dirty Read 발생) |
| READ COMMITTED | 커밋된 데이터만 읽음 |
| REPEATABLE READ | 같은 트랜잭션 내 같은 조회 결과 보장 |
| SERIALIZABLE | 완전 직렬화, 가장 안전하지만 가장 느림 |

#### D — 지속성 (Durability)
커밋된 트랜잭션은 **시스템 장애가 발생해도 영구적으로 반영**되어야 합니다.

```
송금 완료(커밋) 후 서버가 다운되어도
재시작 후 B의 잔액은 증가된 상태로 유지됨
→ WAL(Write-Ahead Logging)으로 보장
```

---

### 3. 커밋 vs 롤백

```sql
BEGIN TRANSACTION;

UPDATE accounts SET balance = balance - 100000 WHERE id = 'A';
UPDATE accounts SET balance = balance + 100000 WHERE id = 'B';

-- 성공 시
COMMIT;

-- 실패 시
ROLLBACK;
```

## 면접 팁

- **"ACID에서 가장 구현하기 어려운 속성은?"** → 격리성(I). 동시성과 성능 사이의 트레이드오프가 있어 격리 수준을 상황에 맞게 선택해야 합니다.
- **"NoSQL은 ACID를 지원하나요?"** → 기본적으로 ACID 대신 BASE(Basically Available, Soft state, Eventually consistent)를 따릅니다. 다만 MongoDB 4.0+ 등 일부 NoSQL은 트랜잭션을 지원합니다.
- **"Dirty Read란?"** → 커밋되지 않은 데이터를 다른 트랜잭션이 읽는 현상입니다. READ UNCOMMITTED 격리 수준에서 발생합니다.
- **백엔드 실무 연관**: Spring의 `@Transactional`은 ACID를 보장하기 위한 어노테이션으로, 격리 수준과 전파 방식을 설정할 수 있습니다.
