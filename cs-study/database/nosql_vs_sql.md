# NoSQL vs SQL

## 질문

SQL(관계형 데이터베이스)과 NoSQL의 차이는 무엇인가요?
각각의 특징, 장단점, 그리고 어떤 상황에서 무엇을 선택해야 하는지 설명하세요.

---

## 핵심 답변

### 1. 기본 개념

| | SQL (RDBMS) | NoSQL |
|-|-------------|-------|
| **정의** | 관계형 모델 기반, 스키마 고정 | 비관계형, 유연한 스키마 |
| **대표** | MySQL, PostgreSQL, Oracle | MongoDB, Redis, Cassandra, DynamoDB |
| **데이터 모델** | 테이블 (행·열) | 문서, 키-값, 컬럼, 그래프 |
| **스키마** | 엄격한 고정 스키마 | 유연(동적) 스키마 |
| **확장 방식** | 수직 확장 (Scale-Up) 주 | 수평 확장 (Scale-Out) 주 |
| **트랜잭션** | ACID 완전 지원 | BASE (최종 일관성), 일부 ACID 지원 |
| **JOIN** | 강력한 JOIN 지원 | 기본 미지원 (데이터 중첩으로 해결) |
| **쿼리 언어** | SQL (표준화) | DB마다 상이 |

---

### 2. NoSQL의 4가지 유형

#### ① 문서형 (Document) — MongoDB, CouchDB
JSON/BSON 형태의 문서 저장. 스키마 자유롭고 중첩 구조 가능.

```json
// MongoDB 문서 예시
{
  "_id": "user_1",
  "name": "철수",
  "address": {
    "city": "서울",
    "district": "강남구"
  },
  "hobbies": ["독서", "코딩"]
}
```

→ 적합: 콘텐츠 관리, 사용자 프로필, 카탈로그

#### ② 키-값형 (Key-Value) — Redis, DynamoDB
단순한 키-값 쌍. 조회가 가장 빠름.

```
SET session:user_1 "{ token: 'abc123', expires: ... }"
GET session:user_1
```

→ 적합: 세션 관리, 캐싱, 실시간 순위표

#### ③ 컬럼형 (Wide-Column) — Cassandra, HBase
행 키 + 컬럼 패밀리. 대용량 쓰기·시계열 데이터에 강점.

→ 적합: IoT 센서 데이터, 로그, 시계열 분석

#### ④ 그래프형 (Graph) — Neo4j
노드와 엣지로 관계를 직접 표현. 관계 탐색이 매우 빠름.

→ 적합: 소셜 네트워크, 추천 시스템, 사기 탐지

---

### 3. ACID vs BASE

#### SQL — ACID
| 속성 | 설명 |
|------|------|
| **Atomicity** (원자성) | 트랜잭션은 전부 성공 or 전부 실패 |
| **Consistency** (일관성) | 트랜잭션 전후 DB 무결성 규칙 유지 |
| **Isolation** (격리성) | 동시 트랜잭션이 서로 영향 없음 |
| **Durability** (지속성) | 커밋된 데이터는 영구 보존 |

#### NoSQL — BASE
| 속성 | 설명 |
|------|------|
| **Basically Available** | 항상 응답은 반환 (오류 대신 오래된 데이터) |
| **Soft State** | 동기화 없이도 시스템이 동작 가능 |
| **Eventually Consistent** | 일정 시간 후 모든 노드가 일관된 상태 |

> 핵심: NoSQL은 **강한 일관성** 대신 **가용성과 성능**을 선택합니다.

---

### 4. CAP 정리 (CAP Theorem)

분산 시스템에서 다음 세 가지를 **동시에 모두 만족할 수 없습니다.**

```
        Consistency (일관성)
              /\
             /  \
            /    \
           /  불가 \
          /________\
Availability      Partition
(가용성)          Tolerance
                  (분단 허용)
```

- **CP 시스템**: 일관성 + 분단 허용 → 일부 요청 거부 가능 (MongoDB, HBase)
- **AP 시스템**: 가용성 + 분단 허용 → 오래된 데이터 반환 가능 (Cassandra, DynamoDB)
- **CA 시스템**: 일관성 + 가용성 → 분산 환경 불가 (단일 노드 RDBMS)

> 실무에서 분산 시스템은 네트워크 분단(P)을 피할 수 없으므로 **CP vs AP를 선택**합니다.

---

### 5. 확장성 비교

#### SQL — 수직 확장(Scale-Up)
서버 자체의 CPU·RAM·디스크를 업그레이드.
- 한계가 명확하고 비용이 기하급수적으로 증가
- 샤딩(Sharding)으로 수평 확장 가능하지만 구현이 복잡

#### NoSQL — 수평 확장(Scale-Out)
서버를 여러 대 추가해 부하를 분산.
- 설계 단계부터 분산을 고려해 쉽게 확장 가능
- 수천 대의 노드로 페타바이트급 데이터 처리 (Cassandra, DynamoDB)

---

### 6. 언제 무엇을 선택할까?

#### SQL을 선택하는 경우
- **데이터 관계가 복잡**하고 JOIN이 많이 필요할 때
- **강한 일관성(ACID)** 이 필수일 때 (금융, 주문, 결제)
- **스키마가 안정적**이고 자주 바뀌지 않을 때
- 복잡한 쿼리·집계·트랜잭션이 필요할 때

```sql
-- 정규화된 관계형 데이터: 주문-결제-재고 원자적 처리
BEGIN TRANSACTION;
  UPDATE inventory SET stock = stock - 1 WHERE product_id = 1;
  INSERT INTO orders (user_id, product_id, amount) VALUES (1, 1, 50000);
  INSERT INTO payments (order_id, status) VALUES (LAST_INSERT_ID(), 'paid');
COMMIT;
```

#### NoSQL을 선택하는 경우
- **대용량 데이터**를 빠르게 처리해야 할 때
- **스키마가 자주 바뀌거나** 데이터 형태가 다양할 때
- **수평 확장**이 필요한 글로벌 서비스
- 단순 읽기/쓰기가 많고 복잡한 JOIN이 없을 때

```javascript
// MongoDB: 사용자마다 다른 필드를 가진 유연한 문서
db.users.insertOne({ name: "철수", age: 25, premium: true, tags: ["dev"] });
db.users.insertOne({ name: "영희", company: "카카오", role: "PM" }); // 다른 필드 OK
```

#### 실무에서는 혼용 (Polyglot Persistence)

```
┌─────────────┐    트랜잭션    ┌──────────────┐
│  PostgreSQL  │ ─────────── │  주문/결제    │
└─────────────┘              └──────────────┘

┌─────────────┐    세션/캐시   ┌──────────────┐
│    Redis     │ ─────────── │  로그인 토큰  │
└─────────────┘              └──────────────┘

┌─────────────┐    콘텐츠     ┌──────────────┐
│   MongoDB   │ ─────────── │  상품 카탈로그│
└─────────────┘              └──────────────┘
```

## 예상 면접 질문

**Q1. NoSQL이 SQL보다 빠른가요?**
> 단순 키-값 조회나 대량 쓰기는 NoSQL이 빠를 수 있지만, **무조건 빠른 것은 아닙니다.** SQL은 인덱스와 쿼리 최적화가 잘 되어 있어 복잡한 조회는 더 빠를 수 있습니다. NoSQL의 속도 이점은 주로 **JOIN 없음, 수평 확장 용이, 스키마 유연성** 에서 옵니다.

**Q2. MongoDB를 사용하면 데이터 정합성 문제가 없나요?**
> MongoDB도 버전 4.0부터 **다중 문서 ACID 트랜잭션**을 지원하지만, 분산 환경에서는 여전히 트레이드오프가 있습니다. 관계형 데이터(외래키, JOIN)가 많다면 MongoDB보다 RDBMS가 적합합니다. MongoDB는 **하나의 문서 내에서의 원자성**만 기본 보장합니다.

**Q3. CAP 정리에서 CA를 선택할 수 없는 이유는?**
> 분산 시스템에서 네트워크 장애(Partition)는 피할 수 없습니다. 네트워크가 분단되면 일관성(C)을 지키려면 일부 노드의 응답을 거부해야 하고(가용성 포기), 가용성(A)을 지키려면 오래된 데이터를 반환해야 합니다(일관성 포기). 따라서 P를 전제로 **CP vs AP 중 선택**합니다.

**Q4. Redis를 데이터베이스로 써도 되나요?**
> 가능하지만 주의가 필요합니다. Redis는 기본적으로 **인메모리** 이므로 재시작 시 데이터가 사라질 수 있습니다. RDB 스냅샷이나 AOF(Append Only File)를 사용하면 영속성을 확보할 수 있지만, 주 저장소보다는 **캐시·세션·실시간 데이터** 용도로 사용하는 것이 일반적입니다.

## 면접 팁

- **"NoSQL이 SQL보다 좋다"** 는 표현은 피하세요. **"트레이드오프에 따라 선택"** 이라는 관점을 보여주는 게 중요합니다.
- **CAP 정리**를 알고 있으면 분산 시스템 설계 이해도를 어필할 수 있습니다.
- 실무 경험에서 "특정 기능은 PostgreSQL, 캐싱은 Redis, 로그는 Elasticsearch를 함께 썼다"처럼 **Polyglot Persistence 사례**를 언급하면 좋습니다.
