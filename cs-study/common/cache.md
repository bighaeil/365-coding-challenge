# 캐시(Cache)

## 질문

캐시(Cache)란 무엇이고, 왜 사용하나요?
메모리 계층 구조, 쓰기 정책과 교체 알고리즘, 그리고 실전에서 발생하는 문제(Cache Stampede, Penetration 등)와 해결 방법을 설명하세요.

---

## 핵심 답변

### 1. 기본 개념

캐시는 **느린 저장소의 데이터를 빠른 저장소에 임시로 복사해두는 것**입니다.
목적은 단 하나 — **접근 속도를 높여 전체 성능을 개선**하는 것.

캐시가 효과를 내는 이유는 프로그램이 **지역성(Locality)** 을 가지기 때문입니다.

- **시간 지역성(Temporal Locality)**: 한 번 접근한 데이터는 곧 다시 접근될 가능성이 높다. (예: 반복문 변수)
- **공간 지역성(Spatial Locality)**: 한 번 접근한 데이터 근처도 곧 접근될 가능성이 높다. (예: 배열 순회)

지역성이 없다면 캐시는 무용지물이며, 오히려 오버헤드만 됩니다.

---

### 2. 메모리 계층 구조 (Memory Hierarchy)

컴퓨터는 여러 속도/크기/비용의 저장 계층을 조합해 사용합니다.

| 계층            | 크기   | 접근 시간 | 특징                    |
| --------------- | ------ | --------- | ----------------------- |
| CPU 레지스터    | B      | < 1ns     | 가장 빠름, 매우 비쌈    |
| L1 캐시         | KB     | ~1ns      | 코어 전용               |
| L2 / L3 캐시    | MB     | ~10ns     | 코어 간 공유(L3)        |
| RAM             | GB     | ~100ns    | 주기억장치              |
| SSD             | TB     | ~100μs    | 보조기억장치            |
| HDD / Network   | TB+    | ms~       | 가장 느림, 가장 저렴    |

**위로 갈수록 빠르고, 작고, 비쌉니다.**
캐시는 이 피라미드에서 "더 빠른 층에 자주 쓰는 것만 올려두는" 전략입니다.

---

### 3. 캐시 쓰기 정책 (Write Policy)

데이터를 **쓸 때** 캐시와 원본(DB/메모리)을 어떻게 동기화할지에 대한 정책입니다.

| 정책              | 동작                                             | 장점                  | 단점                       |
| ----------------- | ------------------------------------------------ | --------------------- | -------------------------- |
| **Write-Through** | 캐시와 원본에 **동시에** 쓴다                    | 일관성 보장           | 쓰기 속도 느림             |
| **Write-Back**    | 캐시에만 쓰고, 나중에 원본에 반영(flush)         | 쓰기 속도 빠름        | 전원 꺼지면 데이터 손실    |
| **Write-Around**  | 원본에만 쓰고 캐시는 건너뜀                      | 자주 안 읽는 데이터에 유리 | 캐시 미스 증가           |

> 중요도 높은 데이터(결제 등)는 Write-Through, 성능이 중요한 로그/세션은 Write-Back이 일반적입니다.

---

### 4. 캐시 교체 알고리즘 (Replacement Policy)

캐시가 꽉 찼을 때 **무엇을 버릴지** 결정하는 알고리즘.
본질적으로 OS의 **페이지 교체 알고리즘**과 같은 문제입니다.

- **LRU (Least Recently Used)**: 가장 오래 사용되지 않은 항목 제거. 가장 널리 쓰임.
- **LFU (Least Frequently Used)**: 가장 사용 빈도가 낮은 항목 제거.
- **FIFO (First In First Out)**: 먼저 들어온 것 제거.
- **Random**: 무작위 제거. 구현이 간단하고 의외로 성능이 괜찮음.

> 💡 **LRU는 면접 단골 구현 문제**입니다.
> HashMap + Doubly Linked List 조합으로 `get`/`put` 모두 O(1)에 구현 가능.

---

### 5. 캐시의 종류 (실전)

#### 🖥️ CPU 캐시
하드웨어 수준에서 동작. 프로그래머가 직접 제어하진 않지만,
**캐시 친화적 코드**(배열 행 우선 순회 등)로 성능이 수 배 차이 납니다.

#### 💾 OS 페이지 캐시
파일 I/O 결과를 RAM에 캐싱. 같은 파일을 두 번째 읽을 때 훨씬 빠른 이유.

#### 🗄️ DB 버퍼 풀
MySQL InnoDB의 **Buffer Pool**, PostgreSQL의 **shared_buffers**.
자주 접근하는 B+Tree 인덱스 페이지를 메모리에 유지해 디스크 I/O를 줄입니다.

#### 🌐 애플리케이션 캐시 — Redis / Memcached
가장 대표적인 패턴은 **Cache-Aside (Lazy Loading)**:

```
[Client] → [App Server] → [Cache] ─hit→ 반환
                             │
                             └miss→ [DB] → Cache에 저장 → 반환
```

#### 🌍 CDN (Content Delivery Network)
이미지/JS/CSS 같은 정적 자원을 사용자와 **지리적으로 가까운** 엣지 서버에 캐싱.
(CloudFlare, CloudFront 등)

#### 🔖 브라우저 캐시
HTTP 헤더 `Cache-Control`, `ETag`, `Last-Modified`로 제어.

---

### 6. 캐시의 문제와 해결

> *"There are only two hard things in Computer Science:
> cache invalidation and naming things."* — Phil Karlton

#### ❶ Cache Invalidation (캐시 무효화)
원본이 변경됐는데 캐시가 옛 값을 반환 → **Stale Data**.
**해결**: 짧은 TTL, 갱신 시 캐시 **삭제**(갱신보다 삭제가 동시성에 안전).

#### ❷ Cache Stampede (캐시 쇄도)
인기 있는 key가 만료되는 순간 수천 개 요청이 동시에 DB로 몰림 → DB 과부하.
**해결**:
- 만료 시간 랜덤화(jitter)
- Mutex 락으로 한 요청만 갱신
- 만료 전 미리 갱신(refresh-ahead)

#### ❸ Cache Penetration (캐시 뚫기)
존재하지 않는 key를 반복 조회 → 매번 캐시 miss → 매번 DB 접근.
**해결**:
- "데이터 없음"도 캐싱 (짧은 TTL의 null 값)
- **Bloom Filter** 로 사전 차단

#### ❹ Hotspot (핫스팟)
특정 key에 트래픽 집중 → 해당 캐시 노드 과부하.
**해결**: 샤딩, 로컬(프로세스 내) 캐시 추가, 복제.

## 예상 면접 질문

**Q1. 캐시와 버퍼의 차이는 무엇인가요?**
> 캐시는 **재사용을 통한 빠른 접근**이 목적이고, 버퍼는 **속도 차이가 다른 장치 간의 일회성 임시 저장**이 목적입니다.
> 예: 디스크에서 읽은 데이터를 RAM에 옮길 때 쓰는 건 버퍼, 같은 데이터를 다시 쓰기 위해 저장해두는 건 캐시.

**Q2. Write-Through와 Write-Back 중 무엇을 쓰시겠습니까?**
> 데이터 중요도와 쓰기 빈도에 따라 다릅니다.
> 금융·결제처럼 유실이 허용되지 않는다면 Write-Through, 로그·세션처럼 성능이 중요하면 Write-Back이 적절합니다.

**Q3. Redis 캐시와 DB의 일관성을 어떻게 맞추시겠습니까?**
> Cache-Aside 패턴에서 **"DB 업데이트 → 캐시 삭제"** 순서를 따릅니다.
> 캐시를 "갱신"하지 않고 "삭제"하는 이유는, 동시 요청 상황에서 오래된 값이 캐시에 덮어쓰일 위험을 줄이기 위해서입니다.
> 또한 TTL을 짧게 설정해 최종 일관성을 보장합니다.

**Q4. LRU 캐시를 O(1)에 어떻게 구현하나요?**
> **HashMap + Doubly Linked List** 조합으로 구현합니다.
> HashMap은 key → 노드 참조를 O(1)에 찾고, Doubly Linked List는 최근 사용 순서를 유지하며 노드 이동/삭제를 O(1)에 처리합니다.
> `get` 시 해당 노드를 리스트 맨 앞으로 이동, `put` 시 꽉 차면 맨 뒤 노드를 제거합니다.

**Q5. Cache Stampede는 무엇이며 어떻게 해결하나요?**
> 인기 key가 만료되는 순간 여러 요청이 동시에 캐시 miss를 일으켜 DB에 몰리는 현상입니다.
> 만료 시간에 랜덤 jitter를 주거나, 락으로 한 요청만 DB에 접근하게 하거나, 만료 전 미리 갱신하는 방식으로 해결합니다.

## 면접 팁

- **"캐시 쓰면 무조건 빨라지나요?"** → 아니요. **지역성이 없으면** 오히려 오버헤드만 발생합니다. 캐시는 공짜가 아닙니다.
- **"캐시 무효화가 왜 어려운가요?"** → 원본과 캐시라는 **두 개의 진실(source of truth)** 이 생기기 때문입니다. 분산 환경에선 동시성까지 겹쳐 더 복잡합니다.
- **cs5(가상 메모리)의 페이지 교체와 본질이 같다**는 점을 언급하면 CS 전반에 대한 이해를 어필할 수 있습니다.
- **cs6(DB 인덱스)의 B+Tree 페이지도 결국 Buffer Pool이라는 캐시에 올라간다**는 점을 연결하면 실무 이해도가 드러납니다.
