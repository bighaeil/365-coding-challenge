# 가상 메모리(Virtual Memory)란 무엇인가요?

## 질문

가상 메모리가 무엇인지 설명하고, 페이징(Paging)과 페이지 폴트(Page Fault)가 어떻게 동작하는지 말씀해주세요.

---

## 핵심 답변

### 1. 기본 개념

**가상 메모리(Virtual Memory)** 란 물리적 RAM이 부족하더라도, 프로세스에게 마치 충분한 메모리가 있는 것처럼 보이게 하는 메모리 관리 기법입니다.

```
프로세스 관점 (가상 주소 공간)     실제 물리 메모리 + 디스크
┌──────────────────┐              ┌──────────────┐
│  0x0000 ~ 0xFFFF  │  ──매핑──▶  │  RAM (일부)   │
│  (전체 다 있는 듯)  │              ├──────────────┤
└──────────────────┘              │  Swap 영역    │
                                  │  (디스크 일부) │
                                  └──────────────┘
```

**핵심 아이디어**: 당장 필요한 데이터만 RAM에 올리고, 나머지는 디스크(스왑 공간)에 두었다가 필요할 때 불러온다.

---

### 2. 페이징(Paging)

가상 메모리를 구현하는 대표적인 방법입니다.

- 가상 주소 공간과 물리 메모리를 **일정한 크기(보통 4KB)의 블록**으로 나눔
- 가상 블록 = **페이지(Page)**
- 물리 블록 = **프레임(Frame)**
- 매핑 정보를 **페이지 테이블(Page Table)** 에 저장

```
가상 주소: [페이지 번호 | 오프셋]
                ↓ 페이지 테이블 조회
물리 주소: [프레임 번호 | 오프셋]

페이지 테이블 예시:
┌────────┬────────┬────────┐
│ 페이지 # │ 프레임 # │ 유효 비트 │
├────────┼────────┼────────┤
│   0    │   3    │   1   │  ← RAM에 있음
│   1    │   -    │   0   │  ← 디스크에 있음 (Page Fault 발생!)
│   2    │   7    │   1   │  ← RAM에 있음
└────────┴────────┴────────┘
```

---

### 3. 페이지 폴트(Page Fault)

CPU가 접근하려는 페이지가 현재 RAM에 없을 때 발생하는 인터럽트입니다.

```
CPU가 가상 주소 접근
        ↓
페이지 테이블 확인
        ↓
유효 비트 = 0?  →  [페이지 폴트 발생!]
        ↓                  ↓
유효 비트 = 1          OS가 처리:
        ↓          1. 디스크에서 해당 페이지 읽기
    물리 메모리         2. 빈 프레임에 로드
    바로 접근          3. 페이지 테이블 갱신 (유효 비트 = 1)
                   4. 프로세스 재시작
```

**페이지 폴트가 잦으면?** → **스래싱(Thrashing)** 발생
페이지 교체가 너무 빈번하여 CPU가 실제 작업보다 페이지 교체에 더 많은 시간을 소비하는 상태.

---

### 4. 페이지 교체 알고리즘

RAM이 꽉 찼을 때 어느 페이지를 디스크로 내보낼지 결정합니다.

| 알고리즘 | 설명 | 특징 |
|---------|------|------|
| **FIFO** | 가장 먼저 올라온 페이지 교체 | 구현 단순, Belady's Anomaly 발생 가능 |
| **LRU (Least Recently Used)** | 가장 오랫동안 사용 안 된 페이지 교체 | 성능 우수, 실제 OS에서 근사 구현 |
| **OPT (Optimal)** | 앞으로 가장 오래 사용 안 될 페이지 교체 | 이론적 최적, 미래를 알아야 해서 실제 불가 |
| **Clock (NRU)** | 참조 비트를 이용한 LRU 근사 | 실제 OS가 많이 사용 |

---

### 5. TLB (Translation Lookaside Buffer)

페이지 테이블 조회를 매번 메모리에서 하면 느리므로, CPU 내부의 캐시(**TLB**)에 최근 변환 결과를 저장합니다.

```
CPU 가상 주소 요청
        ↓
TLB 확인
        ↓
TLB Hit  →  물리 주소 바로 반환 (빠름!)
TLB Miss →  페이지 테이블 조회 (느림) → TLB에 저장
```

## 예상 면접 질문

**Q1. 가상 메모리를 왜 사용하나요?**
> 세 가지 이유가 있습니다.
> 1. **메모리 확장**: 물리 RAM보다 큰 프로세스도 실행 가능합니다.
> 2. **프로세스 격리**: 각 프로세스가 독립적인 가상 주소 공간을 가져 서로의 메모리를 침범할 수 없습니다.
> 3. **효율적 활용**: 당장 필요한 페이지만 RAM에 올리고 나머지는 디스크에 두어 메모리를 효율적으로 씁니다.

**Q2. 페이지 폴트가 발생하면 OS는 어떻게 처리하나요?**
> 1. CPU가 유효 비트 = 0인 페이지에 접근 → **페이지 폴트 인터럽트** 발생
> 2. OS가 제어권을 가져와 해당 페이지를 **디스크(스왑 영역)에서 읽음**
> 3. RAM에 **빈 프레임이 있으면 로드**, 없으면 교체 알고리즘으로 희생 페이지 선택 후 교체
> 4. **페이지 테이블 갱신** (유효 비트 = 1, 프레임 번호 기록)
> 5. 중단됐던 **프로세스 재시작**

**Q3. 스래싱(Thrashing)이란 무엇이고, 어떻게 해결하나요?**
> 페이지 폴트가 너무 빈번하게 발생해 CPU가 실제 작업보다 **페이지 교체에 더 많은 시간을 소비**하는 상태입니다.
> 해결 방법:
> - **워킹셋(Working Set) 모델**: 프로세스가 최근 일정 시간 동안 참조한 페이지 집합만 보장
> - **페이지 폴트 빈도(PFF) 제어**: 폴트율이 높으면 프레임 수 증가, 낮으면 감소
> - **멀티프로그래밍 수준 감소**: 동시 실행 프로세스 수를 줄여 프로세스당 가용 프레임 확보

**Q4. LRU를 OS에서 정확하게 구현하기 어려운 이유는?**
> 완벽한 LRU는 **모든 메모리 접근마다 타임스탬프를 기록**해야 합니다. 그런데 메모리 접근은 매우 빈번하게 발생하므로, 하드웨어 지원 없이 구현하면 오버헤드가 극도로 큽니다. 그래서 실제 OS는 **참조 비트(Reference Bit)를 이용한 Clock 알고리즘** 같은 LRU 근사 방식을 사용합니다.

## 면접 팁

- **"가상 메모리를 왜 쓰나요?"** → 물리 메모리보다 큰 프로세스 실행 가능, 프로세스 간 메모리 격리, 효율적인 메모리 활용이 목적입니다.
- **"페이지 폴트는 항상 나쁜 건가요?"** → 아닙니다. 페이지 폴트 자체는 정상 동작입니다. 너무 빈번하게 발생해 스래싱이 생길 때 문제가 됩니다.
- **"LRU를 OS에서 실제로 구현하기 어려운 이유는?"** → 모든 메모리 접근마다 타임스탬프를 기록하는 것이 하드웨어 비용이 크기 때문에, Clock 알고리즘 같은 근사 방식을 사용합니다.
- **"세그멘테이션과 페이징의 차이?"** → 세그멘테이션은 코드/데이터/스택 등 논리적 단위로 분할(가변 크기), 페이징은 고정 크기로 분할합니다. 현대 OS는 주로 페이징을 사용합니다.
