# 동기/비동기 & 블로킹/논블로킹

## 질문

동기(Synchronous)와 비동기(Asynchronous), 블로킹(Blocking)과 논블로킹(Non-blocking)의 차이를 설명하세요.
네 가지 조합이 어떻게 동작하는지, 그리고 JavaScript의 이벤트 루프와 어떤 관계가 있는지 설명하세요.

---

## 핵심 답변

### 1. 두 개념의 관점 차이

이 두 개념은 흔히 혼동되지만, 바라보는 **관점이 다릅니다.**

| 개념 | 관점 | 핵심 질문 |
|------|------|-----------|
| **동기/비동기** | **작업 완료 통보 방식** | 결과를 누가, 언제 확인하는가? |
| **블로킹/논블로킹** | **호출 중 제어권 반환 여부** | 호출 후 내 코드가 멈추는가? |

> 비유: 카페 주문  
> - **블로킹**: 카운터 앞에 서서 음료가 나올 때까지 기다림 (제어권 없음)  
> - **논블로킹**: 진동벨 받고 자리로 돌아가 다른 일을 함 (제어권 있음)  
> - **동기**: 진동벨이 울렸는지 내가 직접 계속 확인  
> - **비동기**: 진동벨이 울리면 알아서 나에게 알려줌 (콜백/알림)

---

### 2. 블로킹 vs 논블로킹

**제어권(CPU 점유 및 코드 실행 흐름)** 이 핵심입니다.

#### 블로킹(Blocking)
호출한 함수가 작업을 완료할 때까지 **호출자의 실행이 멈춥니다.**

```javascript
// Node.js 동기 파일 읽기 (블로킹)
const fs = require('fs');
const data = fs.readFileSync('file.txt', 'utf8'); // 여기서 멈춤
console.log(data); // 파일 읽기 완료 후 실행
console.log('다음 코드'); // 그 다음 실행
```

```
호출자: ────[요청]────[대기 중... 아무것도 못 함]────[결과 수신]────►
I/O:         ────────────────[처리 중]────────────►
```

#### 논블로킹(Non-blocking)
호출 즉시 **제어권을 돌려받아** 다른 작업을 계속 수행합니다.

```javascript
// Node.js 비동기 파일 읽기 (논블로킹)
const fs = require('fs');
fs.readFile('file.txt', 'utf8', (err, data) => {
  console.log(data); // I/O 완료 후 나중에 실행
});
console.log('다음 코드'); // 요청 직후 바로 실행 ← 여기가 먼저 출력됨
```

```
호출자: ────[요청]────[다른 작업 가능]────[콜백 실행]────►
I/O:         ────────────────[처리 중]────►
```

### 3. 동기 vs 비동기

**결과 처리 시점과 통보 방식** 이 핵심입니다.

#### 동기(Synchronous)
작업 완료 여부를 **호출자가 직접 확인**합니다. 순서가 보장됩니다.

```javascript
// 동기: 반환값으로 결과를 받음 — 내가 확인
function syncTask() {
  const result = doSomething(); // 완료될 때까지 기다렸다가 결과 받음
  return result;
}
```

#### 비동기(Asynchronous)
작업이 완료되면 **시스템이 호출자에게 통보(콜백/이벤트)** 합니다. 순서가 보장되지 않습니다.

```javascript
// 비동기: 콜백/Promise/async-await — 시스템이 알려줌
fetch('/api/data')
  .then(res => res.json())       // 완료되면 시스템이 호출
  .then(data => console.log(data));

console.log('fetch 요청 후 바로 실행'); // 먼저 출력됨
```

### 4. 네 가지 조합

| | 블로킹 | 논블로킹 |
|---|---|---|
| **동기** | ✅ 가장 일반적 | ⚠️ 폴링 방식 (바쁜 대기) |
| **비동기** | ⚠️ 거의 사용 안 함 | ✅ 가장 효율적 |

#### ① 동기 + 블로킹 (Sync Blocking) — 가장 흔한 패턴
결과가 나올 때까지 기다리며, 제어권도 없음.

```javascript
const result = fs.readFileSync('file.txt'); // 멈추고 기다림
```

#### ② 동기 + 논블로킹 (Sync Non-blocking) — 바쁜 대기(Busy Waiting)
제어권은 있지만 완료 여부를 내가 직접 반복 확인. CPU 낭비.

```javascript
// 의사 코드: 완료될 때까지 폴링
let result = null;
while (result === null) {
  result = checkIfDone(); // 직접 계속 확인 → CPU 낭비
}
```

#### ③ 비동기 + 블로킹 (Async Blocking) — 비효율적 조합
완료 통보는 받지만 막아 놓은 상태. 실질적으로 동기 블로킹과 비슷.

```javascript
// select()로 I/O 감시하면서 블로킹하는 패턴 (리눅스 I/O 멀티플렉싱)
// Node.js에서는 거의 마주치지 않음
```

#### ④ 비동기 + 논블로킹 (Async Non-blocking) — 가장 효율적
제어권을 가지고 다른 일을 하다가, 완료 시 시스템이 알려줌.

```javascript
// Node.js의 기본 패턴
fs.readFile('file.txt', 'utf8', (err, data) => {
  console.log(data); // 완료 시 이벤트 루프가 콜백 실행
});
// 요청 후 즉시 다음 코드 실행 가능
```

### 5. JavaScript 이벤트 루프와의 연결

JavaScript는 **싱글 스레드**이지만 논블로킹 비동기로 높은 처리량을 달성합니다.

```
┌─────────────────────────────┐
│         Call Stack           │  ← JS 코드 실행 (싱글 스레드)
└──────────────┬──────────────┘
               │ Web API / libuv에 위임
┌──────────────▼──────────────┐
│    Web APIs / libuv (OS)    │  ← I/O, Timer, Network (멀티 스레드)
└──────────────┬──────────────┘
               │ 완료 시 콜백 등록
┌──────────────▼──────────────┐
│        Callback Queue        │  ← 실행 대기 중인 콜백
│  (Microtask > Macrotask)    │
└──────────────┬──────────────┘
               │ Call Stack이 비면 이동
┌──────────────▼──────────────┐
│         Event Loop           │  ← 큐 → 스택 이동 감시
└─────────────────────────────┘
```

- I/O 요청을 **OS(libuv)에 위임** → JS 스레드는 즉시 다음 코드 실행 (논블로킹)
- 완료되면 **콜백 큐에 등록** → 이벤트 루프가 스택이 빌 때 콜백 실행 (비동기)

```javascript
console.log('1');                        // 동기

setTimeout(() => console.log('2'), 0);  // 비동기 Macrotask

Promise.resolve().then(() => console.log('3')); // 비동기 Microtask

console.log('4');                        // 동기

// 출력 순서: 1 → 4 → 3 → 2
// Microtask(Promise)가 Macrotask(setTimeout)보다 먼저 실행됨
```

## 예상 면접 질문

**Q1. 동기와 블로킹은 같은 말 아닌가요?**
> 다릅니다. **블로킹**은 호출 중 제어권을 뺏기는 것(구현 방식), **동기**는 결과를 내가 직접 확인하는 것(순서 보장 방식)입니다. "논블로킹이지만 동기"인 폴링 방식이 가능하고, "블로킹이지만 비동기"인 select() 패턴도 존재합니다.

**Q2. Node.js는 싱글 스레드인데 어떻게 여러 요청을 동시에 처리하나요?**
> JS 코드는 싱글 스레드지만, I/O 작업은 내부의 **libuv 스레드 풀과 OS 비동기 API**에 위임합니다. JS 스레드는 위임 후 즉시 다음 작업을 처리(논블로킹)하고, I/O 완료 시 이벤트 루프가 콜백을 Call Stack에 올려 실행합니다.

**Q3. async/await은 비동기인데 왜 동기처럼 보이나요?**
> `await`은 Promise가 완료될 때까지 **해당 async 함수의 실행을 일시 중단**하지만, Call Stack에서 해당 함수를 꺼내고 다른 코드가 실행될 수 있게 합니다. 즉, **비동기 논블로킹**이지만 코드 가독성은 동기처럼 보입니다.

**Q4. Promise와 setTimeout의 실행 순서가 왜 다른가요?**
> Promise의 `.then()`은 **Microtask 큐**, setTimeout은 **Macrotask 큐**에 들어갑니다. 이벤트 루프는 Macrotask 하나를 처리한 후 Microtask 큐를 **전부 비운 다음** 다음 Macrotask를 처리합니다. 따라서 Promise가 항상 setTimeout보다 먼저 실행됩니다.

**Q5. 멀티스레드 방식(Java 서블릿) vs 이벤트 루프 방식(Node.js)의 트레이드오프는?**
> **멀티스레드**: CPU 집약적 작업에 유리, 스레드 간 컨텍스트 스위칭 오버헤드, 동기화 문제(race condition, deadlock) 있음.  
> **이벤트 루프**: I/O 집약적 작업에 유리(수만 개 동시 연결 처리 가능), CPU 집약 작업 시 전체 이벤트 루프 블로킹 위험.

## 면접 팁

- **"동기 = 블로킹, 비동기 = 논블로킹"으로 혼동하지 마세요.** 관점이 다르다는 것을 정확히 구분하는 것이 핵심입니다.
- Node.js 면접이라면 **이벤트 루프 + Microtask/Macrotask 순서**까지 설명하면 심화 이해를 어필할 수 있습니다.
- **async/await은 문법적 설탕(syntactic sugar)** 이라는 점, 내부적으로는 Promise와 동일하다는 것을 언급하세요.
- I/O 집약(Node.js 강점) vs CPU 집약(멀티스레드 강점) 트레이드오프를 알면 실무 설계 역량도 보여줄 수 있습니다.
