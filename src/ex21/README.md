# ex21. 창고 재고 이동 시뮬레이션 (Warehouse Stock Move Simulation)

## 문제 설명

`N`개의 창고가 `1`번부터 `N`번까지 있고, 각 창고에는 초기 재고가 들어있습니다.
각 재고에는 **원래 보관되어 있던 창고 번호**가 출처로 표시되어 있습니다.

이후 `M`개의 이동 명령이 순차적으로 주어집니다. 각 명령은 `[from, to, amount]` 형태로,
`from` 창고에서 `to` 창고로 `amount` 만큼의 재고를 이동시킵니다.

이동은 **선입선출(FIFO)** 방식으로 이루어집니다.
즉, `from` 창고에 가장 먼저 들어온 재고부터 순서대로 꺼내서 `to` 창고의 가장 뒤에 쌓습니다.

모든 이동이 끝난 후, 제거 대상 출처 번호가 담긴 배열 `removes`가 주어집니다.
**`removes`에 포함된 번호를 출처로 가지는 재고는 어느 창고에 있든 모두 제거**합니다.

각 창고에 최종적으로 남아있는 재고의 총합을 배열로 반환하세요.

## 입력 조건

- `stocks`: 길이 `N`의 정수 배열. `stocks[i]`는 `i+1`번 창고의 초기 재고량 (출처 = `i+1`)
- `moves`: 길이 `M`의 2차원 배열. 각 원소는 `[from, to, amount]` (1-indexed)
- `removes`: 제거할 **출처 번호** 들이 담긴 배열
- `1 ≤ N ≤ 1,000`
- `1 ≤ M ≤ 10,000`
- `1 ≤ from, to ≤ N`, `from ≠ to`
- `amount`는 이동 시점에 `from` 창고의 총 재고 이하임을 보장
- `removes`의 원소는 `1` 이상 `N` 이하

## 예제

**예제 1**
```
입력:
stocks  = [30, 30, 30, 30]
moves   = [[1, 2, 10], [2, 3, 20], [3, 4, 30]]
removes = [1]
출력: [0, 10, 20, 60]
```

**설명**

| 시점 | 1번 창고 | 2번 창고 | 3번 창고 | 4번 창고 |
| --- | --- | --- | --- | --- |
| 초기 | `[1:30]` | `[2:30]` | `[3:30]` | `[4:30]` |
| `1→2: 10` | `[1:20]` | `[2:30, 1:10]` | `[3:30]` | `[4:30]` |
| `2→3: 20` | `[1:20]` | `[2:10, 1:10]` | `[3:30, 2:20]` | `[4:30]` |
| `3→4: 30` | `[1:20]` | `[2:10, 1:10]` | `[2:20]` | `[4:30, 3:30]` |
| 출처 1 제거 | `[]` | `[2:10]` | `[2:20]` | `[4:30, 3:30]` |
| **합계** | **0** | **10** | **20** | **60** |

## 풀이 아이디어

### 멘탈 모델: 라벨 붙은 택배 상자

재고량을 계산해서 옮기는 게 아니라, **출처 라벨이 붙은 상자를 통째로 들어서 옮긴다**는 감각이 핵심입니다.
상자 안을 들여다보지도, 더하지도 않고 그냥 `to` 창고 맨 뒤에 놓습니다. 라벨(출처)은 절대 건드리지 않습니다.

이동할 때 경우는 딱 두 가지뿐:
- 맨 앞 상자 ≤ 남은 필요량 → **통째로 이동**하고 다음 상자로
- 맨 앞 상자 > 남은 필요량 → **필요한 만큼만 쪼갠 새 상자**를 보내고, 원본은 차감 후 제자리에

### 세 단계

1. **출처 추적** — 재고를 단순 숫자가 아니라 `{ origin, count }` 덩어리로 관리한다. 다른 창고로 옮겨져도 출처 꼬리표는 그대로 유지한다.
2. **FIFO 분할** — 각 창고를 큐로 보고, `amount`를 맨 앞 덩어리부터 꺼낸다. 부족하면 통째로, 넘치면 쪼개서.
3. **Set 기반 필터링** — 모든 이동이 끝난 뒤 `removes`를 `Set`으로 만들어 출처가 제거 대상인 덩어리를 걸러내고 나머지를 합산한다.

## 면접 팁

- **"재고를 어떻게 표현하나요?"** → 숫자 합계가 아니라 `{ origin, count }` 덩어리의 큐로 관리합니다. 출처별 제거 요구사항 때문에 합쳐버리면 정보가 사라집니다.
- **"amount가 맨 앞 덩어리와 안 맞으면?"** → 두 경우뿐입니다. 덩어리 ≤ 필요량이면 통째로 이동, 크면 필요한 만큼만 쪼개고 원본 차감. 어느 쪽이든 출처는 불변입니다.
- **"`shift()`의 문제점은?"** → 앞 원소를 빼면 뒤 원소를 한 칸씩 당기므로 `O(K)`입니다. head 인덱스만 증가시키면 amortized `O(1)`이 됩니다.
- **"덩어리가 무한정 쪼개지지 않나요?"** → 이동 한 번에 쪼개기는 최대 1회만 발생하므로 전체 덩어리 수는 `O(N + M)`으로 묶입니다.
- **설명 흐름**: 출처 추적 → FIFO 분할 → Set 필터링 순으로 설명하고, 마지막에 `shift()` 비용과 head 포인터/deque 최적화를 덧붙입니다.
