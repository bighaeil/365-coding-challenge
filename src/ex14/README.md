# ex14. 전화번호 목록 (Phone Number List)

## 문제 설명

전화번호부에 적힌 전화번호 목록 `phoneBook`이 주어질 때,
어떤 번호가 다른 번호의 **접두사(prefix)인 경우** `false`,
그렇지 않으면 `true`를 반환하세요.

> 카카오 코딩테스트 실제 출제 유형입니다.

## 입력 조건

- `phoneBook`의 길이는 `1` 이상 `1,000,000` 이하입니다.
- 각 전화번호의 길이는 `1` 이상 `20` 이하입니다.
- 같은 전화번호는 중복해서 포함되지 않습니다.

## 예제

**예제 1**
```
입력: phoneBook = ['119', '97674223', '1195524421']
출력: false
설명: '119'가 '1195524421'의 접두사이므로 false
```

**예제 2**
```
입력: phoneBook = ['123', '456', '789']
출력: true
설명: 어떤 번호도 다른 번호의 접두사가 아님
```

**예제 3**
```
입력: phoneBook = ['12', '123', '1235', '567', '88']
출력: false
설명: '12'가 '123'의 접두사이므로 false
```

## 풀이 아이디어

### 방법 1. 완전탐색 — O(n² × m) ❌ 시간 초과
- 모든 쌍을 비교해 startsWith 확인
- 입력이 최대 1,000,000개이므로 시간 초과

### 방법 2. 정렬 + 인접 비교 — O(n log n) ⭐
- 전화번호를 **사전순으로 정렬**하면 접두사 관계인 번호는 반드시 인접하게 됨
- 인접한 두 번호만 비교하면 전체를 확인 가능

```
정렬 전: ['119', '97674223', '1195524421']
정렬 후: ['119', '1195524421', '97674223']
         ↑ 인접! '119'가 '1195524421'의 접두사 → false
```

### 방법 3. 해시셋 — O(n × m)
- 모든 번호를 Set에 저장
- 각 번호의 모든 부분 접두사가 Set에 존재하는지 확인

## 면접 팁

- **"완전탐색이 왜 안 되나요?"** → O(n²)에서 n이 100만이면 연산이 1조 번. 제한 시간 초과입니다.
- **"정렬 후 인접 비교가 왜 동작하나요?"** → 사전순 정렬 시 접두사는 원본 바로 다음에 위치합니다. 'abc'의 접두사인 'ab'는 정렬 후 'abc' 바로 앞에 옵니다.
- **"정렬의 시간 복잡도는?"** → O(n log n × m). m은 번호 길이. 문자열 비교에 O(m)이 필요합니다.
- **카카오 실전 팁**: 입력 크기를 보고 O(n²)이 가능한지 먼저 판단하세요. n이 10만 이상이면 반드시 O(n log n) 이하 풀이를 찾아야 합니다.

# 

**정렬 + 인접 비교 복잡도**

---

**시간 복잡도: O(n log n × m)**

- `sort()` : O(n log n) — n개의 번호 정렬
- 문자열 비교 한 번에 : O(m) — m은 번호 최대 길이
- 합치면 : O(n log n × m)

m이 최대 20으로 고정이므로 사실상 **O(n log n)** 으로 봐도 됩니다.

---

**공간 복잡도: O(1)**

- 추가 자료구조 없음
- 포인터 `i` 하나만 사용
- 단, `sort()` 가 내부적으로 O(log n) 스택을 사용하므로 엄밀히는 O(log n)

---

**완전탐색과 비교**

```
n = 1,000,000 일 때

완전탐색 O(n²): 1,000,000² = 1조 번 연산 → 시간 초과
정렬+비교 O(n log n): 1,000,000 × 20 = 2천만 번 연산 → 통과
```
