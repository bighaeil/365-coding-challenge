# ex22. 이진 트리 최대 경로 합 (Maximum Root-to-Leaf Path Sum)

> **ex13(최대 깊이)의 변형** — "얼마나 깊은가(노드 수)" 대신 "경로 위 val의 합이 얼마나 큰가"를 구한다.

## 문제 설명

이진 트리(Binary Tree)의 루트 노드가 주어질 때, **루트에서 리프까지 이어지는 경로들 중 노드 val의 합이 가장 큰 값**을 반환하세요.

- 경로는 반드시 **루트에서 시작해 리프에서 끝나야** 합니다. (중간 노드에서 멈출 수 없음)
- 빈 트리(`null`)의 최대 경로 합은 `0`입니다.

## 입력 조건

- 노드의 수는 `0` 이상 `10,000` 이하입니다.
- 노드의 값(`val`)은 **정수이며 음수일 수 있습니다.**
- 빈 트리(`null`)의 답은 `0`입니다.

## 예제

**예제 1**
```
        3
       / \
      9  20
         / \
        15   7

입력: root = [3, 9, 20, null, null, 15, 7]
출력: 38        // 경로 3 → 20 → 15
```
> 경로별 합: `3→9 = 12`, `3→20→15 = 38`, `3→20→7 = 30` → 최대 **38**

**예제 2**
```
    1
     \
      2

입력: root = [1, null, 2]
출력: 3         // 경로 1 → 2
```

**예제 3 — 음수 함정**
```
    1
   /
 -5

입력: root = [1, -5]
출력: -4        // 유일한 루트→리프 경로 1 → -5
```
> `1`에서 멈출 수는 없다. `1`은 리프가 아니므로 `-5`까지 내려가야 한다.

**예제 4**
```
입력: root = []
출력: 0         // 빈 트리
```

## 풀이 아이디어

### 핵심 통찰 — ex13에서 딱 한 군데만 바꾼다

`maxDepth`는 "1을 더하며 더 깊은 쪽"을 골랐다. 여기서는 "**내 val을 더하며 더 큰 쪽**"을 고르면 된다.

```
maxDepth(node)   = 1        + Math.max(왼쪽, 오른쪽)
maxPathSum(node) = node.val + Math.max(왼쪽, 오른쪽)
```

재귀 뼈대는 동일하고, 누적하는 값만 `깊이 → val`로 바뀐다.

### ⚠️ 단, 음수 때문에 base case가 다르다

`maxDepth`는 `null`을 `0`으로 두고 끝났다. 하지만 여기서 `null`을 "합 0인 경로"로 취급하면 **한쪽 자식만 있는 노드**에서 문제가 생긴다.

```
[1, -5]  에서  1 + Math.max(maxSum(-5), maxSum(null))
             = 1 + Math.max(-5, 0)
             = 1   ← 오답! (1에서 멈추는 경로는 존재하지 않음)
```

정답은 **리프에서만 경로를 확정**하는 것. 자식이 한쪽만 있으면 그 자식 쪽으로만 내려간다.

```
if (leaf)        return node.val;                         // 경로 끝
if (왼쪽만)      return node.val + maxSum(왼쪽);
if (오른쪽만)    return node.val + maxSum(오른쪽);
else             return node.val + max(왼쪽, 오른쪽);      // 둘 다
```

### 방법 1. 재귀(DFS) — O(n) 시간, O(h) 공간
- 위 4가지 분기로 리프까지 내려가며 최댓값을 끌어올린다.

### 방법 2. 반복(명시적 스택 DFS) — O(n) 시간, O(n) 공간
- 스택에 `[노드, 루트부터의 누적합]`을 함께 담는다.
- **리프에 도달했을 때만** `best`를 갱신한다. 재귀 스택 오버플로우를 피하고 싶을 때 사용.

## 면접 팁

- **"maxDepth와 뭐가 같고 뭐가 다른가요?"** → 재귀 뼈대(`값 + max(좌, 우)`)는 동일. 누적값이 `깊이(1)`에서 `node.val`로 바뀌고, 음수 때문에 **null을 0으로 못 쓰고 리프에서만 확정**한다는 점이 다르다.
- **"음수가 있으면 왜 조심해야 하나요?"** → 없는 자식(null)을 합 0으로 보면 실제로는 존재하지 않는 "중간에서 멈추는 경로"를 만든다. 루트→리프 문제에서는 리프에 도달했을 때만 경로가 완성된다.
- **"공간 복잡도는?"** → 재귀 DFS는 O(h)(균형 O(log n), 편향 O(n)), 반복 스택은 O(n).
- **더 어려운 변형(follow-up)**: "임의의 노드 → 임의의 노드" 경로 최대 합은 **LeetCode 124 (Hard)**. 각 노드에서 "위로 이어붙일 한쪽 값"과 "여기서 꺾이는 전체 경로 값"을 나눠 계산하고, 음수 서브트리는 `Math.max(0, ...)`로 잘라낸다.
