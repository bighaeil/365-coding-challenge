# ex16. 타겟 넘버 (Target Number)

## 문제 설명

n개의 음이 아닌 정수들이 있습니다. 이 정수들을 순서를 바꾸지 않고 **더하거나 빼서** 타겟 넘버를 만들려고 합니다.

사용할 수 있는 숫자가 담긴 배열 `numbers`와 타겟 넘버 `target`이 주어질 때,
숫자를 더하고 빼서 타겟 넘버를 만드는 **방법의 수**를 반환하세요.

> 프로그래머스 Level 2 / 카카오 코딩테스트 유사 유형입니다.

## 입력 조건

- `numbers`의 길이는 `2` 이상 `20` 이하입니다.
- 각 숫자는 `1` 이상 `50` 이하입니다.
- `target`은 `1` 이상 `1000` 이하입니다.

## 예제

**예제 1**
```
입력: numbers = [1, 1, 1, 1, 1], target = 3
출력: 5
설명:
  -1+1+1+1+1 = 3
  +1-1+1+1+1 = 3
  +1+1-1+1+1 = 3
  +1+1+1-1+1 = 3
  +1+1+1+1-1 = 3
```

**예제 2**
```
입력: numbers = [4, 1, 2, 1], target = 4
출력: 2
설명:
  +4+1-2+1 = 4
  +4-1+2-1 = 4
```

## 풀이 아이디어

### 핵심 아이디어
각 숫자마다 `+` 또는 `-` 두 가지 선택지가 있으므로, 모든 경우의 수는 **2ⁿ**가지입니다.
DFS(깊이 우선 탐색) 또는 재귀로 모든 경우를 탐색합니다.

### 방법. 재귀 DFS — O(2ⁿ)

```
numbers = [1,1,1,1,1], target = 3

                     0
               +1 /     \ -1
              1               -1
          +1/  \-1         +1/  \-1
          2     0          0     -2
        ...   ...        ...    ...

리프 노드(깊이 5)에서 값이 3이면 count++
```

1. 현재 인덱스의 숫자를 더하거나 빼서 재귀 호출
2. 인덱스가 배열 끝에 도달하면 현재 합이 target과 같은지 확인
3. 같으면 count +1

## 면접 팁

- **"시간 복잡도는?"** → O(2ⁿ). n이 최대 20이므로 2²⁰ = 약 100만 번 연산으로 충분합니다.
- **"BFS로도 풀 수 있나요?"** → 네. 큐에 `[현재합, 인덱스]`를 저장하며 레벨 단위로 탐색해도 됩니다. 단, 메모리를 더 사용합니다.
- **"완전탐색을 선택한 이유는?"** → n이 최대 20으로 작아서 2ⁿ = 100만 번이 제한 시간 내에 충분히 가능하기 때문입니다. n이 컸다면 DP를 고려했을 것입니다.
- **카카오 실전 팁**: 완전탐색 문제는 먼저 "경우의 수가 몇 개인가"를 계산해서 완전탐색이 가능한지 판단하는 것이 핵심입니다.

# 

**재귀 DFS** 풀이입니다!

---

### 🔍 동작 흐름 (`[1,1,1,1,1]`, target=3)

```
dfs(0, 0)
  ├─ dfs(1, +1)
  │    ├─ dfs(2, +2)
  │    │    ├─ dfs(3, +3)
  │    │    │    ├─ dfs(4, +4) → dfs(5, +5) dfs(5, +3) ✓
  │    │    │    └─ dfs(4, +2) → dfs(5, +3) ✓  dfs(5, +1)
  │    │    └─ dfs(3, +1) → ...
  │    └─ dfs(2, 0) → ...
  └─ dfs(1, -1) → ...

총 5번 count++ → 5 반환
```

### 💡 핵심 포인트

- `dfs` 내부에 `count`를 클로저로 참조 — 별도 파라미터 없이 깔끔하게 집계
- 리프 노드(`index === length`)에서만 정답 체크
- 매 노드에서 `+`, `-` 두 갈래로 분기 → 트리 깊이 n, 총 노드 수 2ⁿ

### 📊 복잡도

| | |
|--|--|
| 시간 | O(2ⁿ) — n 최대 20 → 약 100만 번 |
| 공간 | O(n) — 재귀 호출 스택 깊이 |

# BFS 풀이

재귀 대신 **큐**를 사용해 레벨(인덱스) 단위로 탐색합니다.

### 아이디어

- 큐에 `[index, sum]` 상태를 저장
- 큐에서 하나 꺼낼 때마다 현재 숫자를 `+`, `-` 한 두 상태를 다시 큐에 넣음
- `index === numbers.length`에 도달한 상태 중 `sum === target`인 것을 count

```javascript
function solution(numbers, target) {
  let count = 0;
  const queue = [[0, 0]]; // [index, sum]

  while (queue.length > 0) {
    const [index, sum] = queue.shift();

    if (index === numbers.length) {
      if (sum === target) count++;
      continue;
    }

    queue.push([index + 1, sum + numbers[index]]);
    queue.push([index + 1, sum - numbers[index]]);
  }

  return count;
}
```

### 🔍 동작 흐름 (`[1,1,1,1,1]`, target=3)

```
큐: [0,0]
 → [1,+1] [1,-1]                    (0번 숫자 처리)
 → [2,+2] [2,0] [2,0] [2,-2]        (1번 숫자 처리)
 → ...                              (레벨마다 상태 수가 2배씩 증가)
 → index=5인 상태 32개 중 sum=3인 상태 5개 → count=5
```

### 💡 DFS와의 차이

- **탐색 순서**: DFS는 한 경로를 끝까지 파고들지만, BFS는 같은 인덱스(레벨)의 상태를 먼저 모두 처리
- **정답 개수는 동일**: 리프 상태(`index === length`)를 전부 확인하므로 결과는 DFS와 같은 `5`

### 📊 복잡도

| | |
|--|--|
| 시간 | O(2ⁿ) — DFS와 동일하게 모든 상태 탐색 |
| 공간 | O(2ⁿ) — 큐에 마지막 레벨의 상태(최대 2ⁿ개)가 동시에 쌓임 |

> **면접 팁**: 이 문제는 정답 경로의 "최단 거리"를 찾는 게 아니라 "모든 경우의 수"를 세는 문제라, BFS의 레벨 탐색 이점이 없고 오히려 큐 때문에 메모리만 O(2ⁿ)로 늘어납니다. 그래서 **재귀 DFS가 더 적합**합니다. `shift()`의 O(n) 비용을 피하려면 포인터 인덱스나 별도 큐 자료구조를 쓰는 것도 언급하면 좋습니다.
